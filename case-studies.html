The customer

A global leader in entertainment, gaming and hospitality.

Every minute a customer waits for support is a minute of brand damage in a business that runs on real-time experience.

The team had a complex troubleshooting process: agents working through a long checklist, querying multiple internal systems, escalating to engineering when it didn't fit a known pattern. Resolution averaged around two hours.

They came to WSO2 with a question: can AI agents actually move the needle on this or will it just be a chatbot bolted on?

What we built

An AI Troubleshooting Agent inside a multi-agent architecture.

Not a single chatbot. The system the FDE team built combined a specialised troubleshooting agent with peer agents that handle escalation, compliance and human handoff.

Troubleshooting Agent

The specialist. Tool-using agent that queries the systems an experienced support engineer would query in parallel then reasons about the result.

Multi-agent orchestration

Peer agents handle escalation paths, compliance checks and routing back to humans when ambiguity exceeds confidence thresholds.

Identity-governed access

Agent ID for every agent. Fine-grained permissions on which tools and data each agent could touch auditable, revocable, governed.

API & MCP integration

WSO2 API Platform and AI Gateway exposed the right enterprise APIs to the agents and governed which LLM endpoints the agents could call.

Observability

Every agent decision logged, every tool call audited, every escalation traced so the team could keep improving the system after we left.

Capability transfer

Their engineers paired on every component. By go-live, the in-house team owned the architecture, the prompts, the evaluation harness and was already shipping the next agents.

A fixed-scope engagement would have shipped a chatbot.

An FDE engagement structured around the outcome "cut the time our customers wait for a resolution" kept adjusting the architecture, the tool choices, and the workflow until the two-hour number actually moved.

The multi-agent design, the integration points with their existing systems, and the governance around the agents all came out of iteration with the customer's engineers not out of a pre-written SOW.
